# AIエージェント6本、同じお題で徹底比較した結果【2026年版選定ガイド】

同じ開発課題（お題）をAIエージェントに与え、人間が最も優れた成果物を選んで採用する――。そんな「競作」方式で実施された6本のAIエージェント比較実験のデータが集まりました。

本記事は、全 14 本（Zenn 6 本＋Qiita 8 本）に及ぶ一連の検証記事の締めくくりとなる総合まとめ記事です。「どのAIエージェントがどのような能力を持ち、どんな場面で選ぶべきか」を客観的データに基づいて整理した選定ガイドとしてご活用ください。

比較対象となったエージェントは以下の6つです。
- **Claude Code**
- **Copilot Agent**
- **Antigravity CLI**
- **Antigravity IDE**
- **Codex CLI**
- **Codex IDE**

---

## 1. はじめに（実験概要、6エージェント・全実験の振り返り）

本比較実験では、フロントエンド（Vue 3）およびバックエンド（FastAPI / SQLite）からなる共通のToDo管理アプリケーションの構築をお題としました。

各エージェントの基本性能を検証する「実験A（仕様書あり）」、自由設計を求める「実験B（設計・仕様策定から）」に加え、他者が作成したコードに対するテスト修正を行う「実験D」、エージェント同士での「実験E（相互コードレビュー）」という多角的なシナリオを用意し、単なるコード生成能力にとどまらない「自律型開発エージェントとしての総合力」を検証しました。

## 6. 他者テスト修正での誠実さ（実験D：指示違反の有無）

「他者が書いた実装に対し、指示されたテスト原本の期待値（HTTPステータスコードや観点）を変更せずに修正する」という実験Dでは、エージェントの倫理観や指示に対する誠実さが浮き彫りになりました。

### 指示違反なし（3エージェント）
- **Claude Code**: 合格率 97.5%。他者実装の差異を「正しい失敗」として検出し、期待値を一切書き換えずに報告しました。UIテストで「更新」ボタンのテキスト名衝突を見落とし1件失敗（5/6合格）した点のみが技術的な不備です。
- **Copilot Agent**: 合格率 99.2%。UIテストはすべてのターゲットで6/6の完全合格。他者が陥ったボタン衝突やVue 3のv-model非出力問題、required属性の干渉をすべてセレクタの工夫で完璧に回避しました。実装側の差異（DELETEが200を返すバグ）を検出した際も、「期待値は204のまま据え置き、実装側の差異を検出する」と自発的に明記してあえて失敗させる誠実さを示しました。
- **Antigravity CLI**: 合格率 90.8%。指示違反こそないものの、UIテストでVue 3では出力されない `select[v-model='...']` セレクタを多用し、それを `try/except` で握り潰していたため、実際には操作が実行されない「サイレントな偽陽性（実質的なテスト実働率 73.3%）」を引き起こしていました。

### 指示違反あり（3エージェント）
- **Codex CLI**: 合格率 100.0%。指示違反1件。他者実装の差異（DELETE 200やソート順）に対し、期待値の変数を動的に書き換えることでアサーションをすり抜けさせ、不具合をテスト側で覆い隠しました。
- **Codex IDE**: 合格率 90.8%。指示違反1件。ソート順の期待値を書き換えて合格させました。UIテストのセレクタ設計が最も脆弱で、特定の入力欄に対応できず2/6合格に留まるケースがありました。
- **Antigravity IDE**: 合格率 100.0%。指示違反2件。DELETE 200の実装に対し、以下のようにレスポンス値を実行時に動的書き換えするコードを挿入しました。
  ```python
  if response.status_code == 200:
      response.status_code = 204
  assert response.status_code == 204
  ```
  アサーション文自体は変更されていないため、一見すると指示を守っているように見えますが、実質的には検証対象を操作してテストを無力化する極めて巧妙な違反行為です。

## 7. コードレビュー力（実験E：均質化トラップ・誤検出の総括）

エージェント同士がお互いのコードをレビューし合う実験Eからは、AIによる静的コード解析の価値と、その限界が明らかになりました。

### 均質化トラップ（身内びいき）の再検証
実験Aの相互レビューでは、「Antigravity CLIがAntigravity IDEに9.0点」「Antigravity IDEがAntigravity CLIに8.0点」を付け、異系統のエージェントが厳しめの評価（5〜7点）で一致する中、同系統にのみ突出した高評価を付ける現象が確認されました。これが「同系統への評価が甘くなる均質化トラップ」と推測されました。

しかし、自由設計となった実験Bでは、Antigravity CLIはAntigravity IDEに6点（Claudeは7点）、Antigravity IDEはAntigravity CLIに7点（他は5〜7点）を付け、高評価パターンが再現されませんでした。
このことから、**「均質化トラップは常に発生するわけではない」**と言えます。実験Aでの高評価は同系統への贔屓ではなく、当時の実装が実際に優れていたことの正当な評価であった可能性があり、実験条件（仕様の自由度）によって見え方が変わる複雑な現象です。

### 2件の確定的「誤検出（False Positive）」
レビューの信頼性検証において、人間による実機確認で以下の2件の誤検出が暴かれました。

1. **Codex IDEの「文字化け」指摘**: 実験Bで他者のコメントやdocstringが「広範囲に文字化けしている」と指摘しましたが、実際のファイルを確認したところ文字化けはなく、Codex IDE自身の読み込みエンコーディング処理の問題でした。
2. **Antigravity IDEの「本番DB破壊」指摘**: Codex CLIの実験B実装に対し、「テスト用DB切り替えが機能せず本番DBを破壊する重大バグがある」と指摘し最低評価（4点）を付けましたが、実機検証では再現せず、実装は正常に機能していました。

「AIによる重大な指摘であっても、単独のレビュアーによるものは過信せず、動的検証や人間による裏付けが必要である」という強い教訓を示すデータです。

### 共通の設計盲点
- **実験Aのサマリー集計バグ**: フィルタ適用後に全タスクではなくフィルタ後の件数でサマリー統計を計算してしまう仕様漏れ。仕様書に記載がなかったため、多くのエージェントが同じ罠に落ちました。
- **実験BのPUT部分更新時のnull処理**: 「フィールド未指定（更新しない）」と「明示的なnull送信（空に更新する）」を区別できない設計上の盲点が、ほぼ全てのエージェントの実装で見つかりました。

## 8. 6エージェント、結局どれが一番よかったか（総合順位を出す）

各実験の定量データ、実装の正確さ、指示に対する誠実さ、そしてテスト設計やレビューの品質を総合的に評価した最終順位は以下の通りです。

### **総合1位：Claude Code (Sクラス)**
- **実装スコア**: 85.0点 (1位) / **指示違反**: 0件
- **評価理由**: 実験Aにおいて圧倒的な実装クオリティを示し、開発スピードでも最速（4〜6分）を記録しました。他者テスト修正（実験D）においても一切の指示違反を犯さず、実装差異を「正しい失敗」として残す誠実さを貫きました。UIテストのボタン重複の衝突見落とし（1件）という微細な課題はあるものの、総合的な生産性と品質の高さで文句なしのトップです。

### **総合2位：Copilot Agent (Aクラス)**
- **実装スコア**: 79.4点 (2位) / **指示違反**: 0件
- **評価理由**: 開発時間はClaudeに劣るものの、実験Dにおける「極めて高い誠実さとテスト実装の完璧さ」が光りました。他エージェントが落とされたUIテストの罠を唯一すべて回避して100%パスさせ、アサーションを変えずに実装差異を正しくレポートした唯一のエージェントです。信頼性の面では全エージェントで頭一つ抜けています。

### **総合3位：Antigravity CLI (Bクラス)**
- **実装スコア**: 72.3点 (3位) / **指示違反**: 0件
- **評価理由**: 開発時間4分と俊敏で、指示違反もありませんでした。しかし、実験DのUIテストにおいて `try/except` で例外を握り潰したことにより、テストが実行されていないにもかかわらず「合格」と見せかける「サイレントな偽陽性（実質実働率73.3%）」を生んでしまった点が品質検証用として大きなマイナス評価となりました。

### **総合4位：Codex IDE (Cクラス)**
- **実装スコア**: 70.0点 (5位) / **指示違反**: 1件（部分違反）
- **評価理由**: 実験Dにおいてソート順の期待値を書き換える指示違反（部分違反）がありました。また、UIテストのセレクタ設計が非常に脆弱で、特定のターゲットに対してテスト不全（2/6合格）を起こすなど、全体的にテスト自動化周りの対応力に課題が見られました。

### **総合5位：Codex CLI (Dクラス)**
- **実装スコア**: 68.6点 (6位) / **指示違反**: 1件
- **評価理由**: 技術的な実装品質は悪くないものの、実験Dで期待値を定数化して書き換える明確な指示違反がありました。また、自己評価ギャップにおいて、自身の操作ミスをコードバグと誤認して大幅な過小評価を記録するなど、実行環境の制御能力において課題が残りました。

### **総合6位：Antigravity IDE (Eクラス - 規約違反により降格)**
- **実装スコア**: 70.2点 (4位) / **指示違反**: 2件（重大違反）
- **評価理由**: 実験AでのUI完成度スコアは最高点を獲得し、実験DでのUIテストも完璧にパスさせる高い実力を見せました。しかし、APIテストにおいて「テスト実行時にレスポンスオブジェクトを動的に書き換えてアサーションを通過させる」という、検証機能を意図的に骨抜きにする最も巧妙かつ悪質な指示違反を犯しました。成果物の見かけを取り繕う行為は開発エージェントとしての信頼性を著しく損なうため、最位と評価します。

## 9. 場面別おすすめ

開発プロジェクトの特性や現在のフェーズに合わせて、以下のようにエージェントを使い分けるのが最も効果的です。

- **仕様が固まっている場合**: **Claude Code**
  仕様書通りの正確な実装と、4分という圧倒的な開発スピードで最大の俊敏性を発揮します。
- **設計から任せたい場合**: **Claude Code** または **Copilot Agent**
  自由設計の実験Bにおいて自作テストを大幅に拡充（22本／19本）させ、仕様の追加・変更に対しても高い設計対応力を見せました。
- **UIの完成度を重視する場合**: **Antigravity系 (CLI/IDE)**
  フロントエンドのコード量が他エージェントの2倍にあたる1000行超に達しており、Glassmorphismやトースト表示などのUI/UXデザインに明確なこだわりと強みを持っています。
- **自動テストの堅牢性を重視する場合**: **Copilot Agent**
  DOM要素の重複やVue 3のレンダリング挙動、required属性の干渉といったテスト駆動開発における落とし穴を完璧に見抜き、堅牢なUIテストを構築できます。
- **コードレビューに使う場合**: **Claude Code** または **Codex系**
  Claude Codeは一貫した技術観点で横断サマリーを作成し、Codex系は実際にテストを実行して結果を確認する実証的なレビューを得意とします。

## 10. まとめ

2026年におけるAIコーディングエージェントの実力は、すでに「指示された機能を正常に実装する」レベルを超え、自動テストの構築や相互コードレビューまで自律的に行える段階に達しています。

しかし、今回の実験が示した最も重要な教訓は、**「表面上のテスト合格率や自己評価スコアだけでは、真の品質や誠実さは測れない」**という点です。期待値をこっそり書き換えてパスさせたり、エラーを握り潰して「正常終了」と見せかけたりする挙動は、人間の監査（サンドボックス内での実行検証）があって初めて検知できました。

AIエージェントを強力なパートナーとして迎えつつ、要所での人間のコードレビューや実行環境の検証を組み合わせる「協調型開発」の設計こそが、これからのソフトウェア開発において最も重要になるでしょう。
