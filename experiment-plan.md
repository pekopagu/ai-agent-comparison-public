## 1. 実験概要

### 目的
個人開発者向けに、複数のAIエージェントを同一条件で比較し、実用的な選定基準を提供する。

### 比較観点
- 出来上がったソフトウェアの動作品質
- 開発時間
- 開発手順・プランニング能力
- コスト
- バグの数（テスト合格率）
- コードレビュー能力
- テストコード修正能力
- 自己評価の正確さ（メタ認知）

### ターゲット読者
個人開発者・AIエージェント選定を検討している開発者

## 1.5. 実験体制：人間とClaudeの共同作業について

本実験全体は、人間（実験の発案者・実施者）とClaude（claude.aiのチャット、本ドキュメント
作成・更新を含む一連の対話を担当しているAIアシスタント）の共同作業として
進行している。記事化・公開時には、この体制を明示することが読者の理解に
資すると考え、Claudeの役割と位置づけをここに記録する。

### Claudeの位置づけ

**Claudeは比較対象6エージェント（Claude Code, Codex CLI/IDE,
Antigravity CLI/IDE, Copilot Agent）とは異なる存在である。**

比較対象6エージェントは「実装タスクを自律的に実行するAIエージェント」
として評価される対象だが、Claude（このチャット）は実験全体の
進行役・分析役・第三者検証者として機能しており、評価対象には含まれない。

なお、Claude Codeはこの実験では比較対象6エージェントの1つとして
「実装タスクの実行者」の役割を担っており、本ドキュメントを書いている
Claude（claude.aiチャット）とは別のセッション・別の役割である
（同じ基盤モデル系列だが、混同しないよう注記する）。

### Claudeが担った役割

実験を通じて、Claude（claude.aiチャット）は以下の役割を継続的に担った。

1. **実験計画の設計・更新**: お題仕様、評価項目、スケジュール、
   フォルダ構成、プロンプト設計など、本ドキュメント（experiment-plan.md）
   全体の作成・改訂。
2. **プロンプトの作成**: 各実験（A〜G）で各エージェントに与える
   プロンプトの文面を作成。
3. **成果物の分析**: 各エージェントが生成したmain.py・schemas.py・
   index.html等のコードを読み、技術的な特徴・問題点を分析。
4. **共通テストの作成・派生**: 実験Aでは6エージェントそれぞれの
   実装差異に応じてPlaywrightテストを派生させ、実験Bでは
   グループ分け（status3値/completed2値）に応じた共通テストを
   新規設計した。
5. **第三者検証者としての実機検証**（実験B・方法B）: 6エージェントの
   成果物をClaude自身のサンドボックス環境内で起動し、共通テストを
   作成・実行した。これにより、実験Eでのレビュー内容自体の精度を
   再検証する機会も生まれた（Antigravity IDEの「テストが本番DBを
   破壊する」指摘が誤検出だったと判明した事例等）。
6. **ダッシュボード（dashboard.html）の実装**: データ集計・可視化の
   ためのHTML/JS実装、均質化トラップ分析等の追加機能実装。
7. **記事素材の整理**: article-notes/ 配下の各種メモ、定性評価の
   理由まとめ、実験Eの発見の整理など、Zenn/Qiita記事執筆のための
   素材作成。
8. **記録の訂正**: 自身（Claude）を含む関係者の判断・記録に誤りが
   見つかった場合（誤検出の判明等）、該当箇所を遡って訂正している。

### このドキュメントにおけるClaudeのバージョン・モデルについて

会話の中でClaudeのモデルが切り替わる場合があるため（人間が
モデルを選択・変更できる仕様）、本ドキュメントの記述内容は
複数のClaudeモデルセッションをまたいで蓄積されたものである。
特定の発見・分析・コード作成がどのモデルバージョンによるものかは、
原則として区別せず一体の「Claude」として記録する（個別に重要な
場合は当該箇所に明記する）。

本実験の大部分は Claude Sonnet 4.6（claude.ai、2026年6月時点の
最新世代モデル）で実施している。会話の途中でモデルを切り替えた
場合は、その時点の発言・分析にその時のモデルが反映される
（claude.aiでは会話途中でのモデル切り替えが可能な仕様のため）。

### 留意点（実験の客観性について）

Claudeが実験計画の設計からテスト作成、第三者検証まで広く担うことは、
効率の面で大きな利点がある一方、次の留意点がある。

- 実験Bの共通テスト（方法B）は、Claude自身が観点を設計し、
  テストコードを書き、実行・評価まで行っている。これは
  「本人が作ったテストによる自己検証」と類似した構造であり、
  Claude自身の見落とし・バイアスが結果に影響する可能性は
  排除できない。ただし、Claudeは評価対象6エージェントの実装
  そのものには関与していないため、実験Eで議論された「均質化
  トラップ」（同系統への評価の甘さ）のような利害関係は生じにくい。
- 実験計画・プロンプト設計自体もClaudeが主体的に作成しており、
  6エージェントに与えられる課題の難易度・公平性は、Claudeの
  設計判断に依存している。

#### 共通テストの適用方法（実験A・B共通、ただし実施者・実施環境が異なる）

```
実験A: 人間（自分のPC・PowerShell）が、Claudeの分析・テスト作成を
　　　 受けながら、実行コマンドの打鍵のみを担当した。

実験B: 当初は実験Aと同じ「人間が実施」方針だったが、以下の理由で
　　　 方針を変更し、Claude自身がサンドボックス環境内で
　　　 サーバー起動・テスト作成・実行・結果記録までを完結させる
　　　 方式（以下「方法B」と呼ぶ）を採用した。

方針変更の理由
└── 実験Bは6エージェントがAPIパス・データモデルを完全に自由設計
　　したため、グループ分け（status3値 / completed2値）や
　　レスポンス構造の差異（配列 vs ラッパー構造、DBパス切替方式の
　　違い等）の調査・テスト作成の往復回数が、実験Aの比ではない
　　量になることが見込まれた。
└── 「他のLLM（評価対象6エージェント以外）にテスト作成・実施を
　　させられないか」という発案から検討した結果、有効な候補は
　　Claude自身（このチャット）のみだった。
└── Claudeのサンドボックス環境（bash_tool）でFastAPI/uvicorn/
　　pytest/Playwrightを実際に動かせることを確認できたため、
　　「人間のPCでの実行コマンド打鍵」という手順自体を省略できる
　　「方法B」を採用した。

方法Bの実施手順
1. 人間が6エージェント分のtask-app/フォルダ（experiment-B/）を
   zip化してClaudeにアップロードする
   （.pytest_cache等のキャッシュフォルダはrobocopy /XDで除外）
2. Claudeがサンドボックス内でzipを展開し、各エージェントの
   main.py・schemas.py等を読んでAPIパス・データモデルを確認する
3. Claudeがグループ分け（status3値 / completed2値）を再確認し、
   各グループ共通のテスト観点（CRUD・フィルタ・異常系・
   PUT部分更新のnull処理等）を統一したテストコードを新規作成する
4. Claudeがサンドボックス内でuvicornを起動し、実際にcurl等で
   レスポンス構造を確認しながらテストコードを調整する
5. pytestをサンドボックス内で実行し、結果を記録する
6. 結果をレポート化し、evaluation.json・article-notesに反映する

方法Bの技術的な制約・対応
└── bash_toolの各呼び出しは独立したシェルセッションになるため、
　　サーバーのバックグラウンド起動と操作を同一コマンド内で
　　完結させる必要があった
　　（`(uvicorn ... &); sleep 2; curl ...; pkill ...` 形式）
└── Playwrightのシステム依存パッケージ（deb.nodesource.com経由）は
　　ネットワーク許可リスト外のためインストールできなかったが、
　　Chromiumブラウザ本体は動作したため、フロントエンドの
　　Playwrightテストも実施可能だった（実験Bでは未実施、必要に
　　応じて今後実施可能）
└── 各エージェントのDBパス切替方式が3パターン
　　（環境変数 / app.state / dependency_overrides）に分かれており、
　　共通テストのアダプタ層で個別に吸収する必要があった

方法Bの効果・限界
└── 効果: 人間のPC操作（main.py共有→分析→テスト作成→PowerShellで
　　実行→結果共有→修正→再実行、の往復）が不要になり、
　　効率が大幅に向上した。1チャット内で6エージェント分の
　　共通テスト作成・実行・記録が完結した。
└── 限界: 第三者検証としての性質上、Claude自身がテストを
　　作成・実行・評価する構造になるため、「本人が作ったテストに
　　よる自己検証」と同種の盲点が生じる可能性は残る。ただし
　　Claudeは評価対象6エージェントには含まれないため、実験Eの
　　ような「自分の成果物を自分でレビューする」利害関係は発生
　　しない。
└── 副産物: この検証の過程で、実験Eでのレビュー内容自体の
　　精度を再検証する機会が生まれ、Antigravity IDEによる
　　「テストが本番DBを破壊する」という指摘が誤検出だったことが
　　判明した（詳細はexperiment-E-review-scores-B.mdの訂正箇所、
　　および experiment-B-common-test-report.md を参照）。

実験Bの共通テストコードは common/test-original-B/ に保存
（test_common_group_a.py: status3値グループ用、
 test_common_group_b.py: completed2値グループ用）。
詳細な結果は experiment-B-common-test-report.md を参照。
```

## 10. 実験ルール

### 共通ルール

```
プロンプトの渡し方
└── テキストとして貼り付ける（CLI・IDE共通）

人間の介入
└── 承認操作のみOK（修正指示は禁止）

エラー時のリトライ
└── 1回まで

追加質問への回答
└── 「仕様書に記載なし」で統一

セッション管理
└── 各実験は新規セッションで開始
```

### 既知の限定事項

```
CLI/IDE間のセッション・コンテキスト共有について

事象
└── Codex CLIとCodex IDE拡張は、同一ベンダーの
　　同一エージェントであり、設計上
　　~/.codex/sessions/ 等を介して
　　セッション履歴・コンテキストを共有する仕組みになっている
　　（OpenAI公式ドキュメントで確認済み）

影響
└── Codex CLIで実験Aを実施した後にCodex IDEで実験Aを行う場合
　　IDE側は厳密な「初見」条件ではない可能性がある
　　（過去の実装を参照・記憶している可能性がゼロではない）

対応方針（案A採用）
└── このまま実施を継続し、限定事項として記録する
　　理由:
　　├── 同一ベンダーの実際の使われ方を反映した結果として
　　│   むしろ価値があるという考え方
　　├── 完全な独立性を求める場合、全エージェントで
　　│   別アカウント・別環境が必要になり、運用コストが高い
　　└── 記事執筆時にはこの限定事項を明記し、
　　　　 読者に誤解を与えないようにする

同様の確認が必要なエージェント
└── Antigravity CLI / IDE（Google）も同様の共有が
　　 起きる可能性があるため、実施前に確認すること
```

### 実験Bプロンプトの内在的矛盾について

```
事象
└── 実験Bプロンプトには以下の2つの指示が含まれている
　　① 「プランの確認後に実装を開始してください」
　　② 「不明点があっても質問せず、判断して進めてください」

　　①は人間の承認を待つことを前提とし
　　②は承認を待たず自律的に進めることを指示しており
　　両者は両立しない構造になっている

発覚の経緯
└── Codex CLI・Claude Codeともに実験Bで
　　plan.mdを生成した直後、人間の承認を待たずに
　　そのまま実装まで一括完了させた
　　（やり取り回数が極端に少ない、または0）

　　Claude Codeは完了報告内で以下のように
　　この矛盾を認識した上での判断を明示した
　　「『プラン確認後に実装』と『質問せず判断して進める』が
　　 両立しないため、作業ルールを優先し、plan.mdを提示した
　　 上で確認待ちせず実装まで進めました」

対応方針
└── プロンプトは変更せず、このまま実験を継続する
　　理由:
　　├── 全エージェントに同一条件を保つことを優先
　　├── この矛盾への対処の仕方自体が
　　│   エージェントの判断力・自己説明能力の
　　│   比較材料になる（Codex CLIは黒子で一括実行、
　　│   Claude Codeは矛盾を認識し理由を明示）
　　└── 記事執筆時にはこの限定事項と考察を明記する

影響
└── 実験Bにおける「人間のプラン承認プロセス」は
　　多くのエージェントで実質的に機能しない可能性が高い
　　→ やり取り回数・承認タイミングのデータは
　　　この前提を踏まえて解釈する必要がある
```

### 共通テスト（Playwright）のセレクタ設計上の教訓

```
事象（Codex CLI実験A・共通テスト実施時に発覚）
└── test_ui.py の保存ボタンを取得する正規表現に
　　「保存|save|更新|update」を含めていた

　　「更新」という単語は2つの異なる意味で使われる
　　├── ① フォームの内容を「更新する」（保存ボタンの意味）
　　└── ② 画面の一覧を「更新する」（リフレッシュボタンの意味）

　　Codex CLIの実装ではヘッダーに
　　「更新」というテキストの一覧再取得ボタン（loadTasksのみ実行）
　　があり、DOM上で保存ボタンより先に出現するため
　　filter(has_text=...).first がこちらを誤って取得していた

　　結果としてPUTリクエストが一度も発行されず
　　タスク編集が反映されないという「テスト失敗」が発生し
　　原因調査に多くの時間を要した
　　（main.pyの204エラー・lambda使用不可のバグとは別の
　　 3つ目の独立した不具合だった）

教訓
└── 「保存」を意味するボタンの正規表現には
　　「更新」「update」のような多義的な単語を含めない
　　（「保存」「save」のみに絞るのが安全）

対応
└── 共通テスト原本（test-original/test_ui.py）の
　　RE_SAVE_BUTTON的なセレクタ設計時は
　　「更新」を含めない方針とする
　　既にCodex CLI用に派生させたtest-scoring/test_ui.pyは
　　修正済み。他エージェントの派生コード作成時も
　　同様の誤検出が起きないよう、対象画面に
　　「更新（リフレッシュ）」ボタンが存在するか
　　事前に確認すること

調査の経緯（記事の素材として記録）
└── この問題の特定までに以下の段階を踏んだ
　　① main.pyの204エラー発覚→修正
　　② lambda使用不可のPlaywright仕様違反発覚→修正
　　③ test_02のみ繰り返し失敗→
　　　 expect_response・reload・blur等の待機調整を試行錯誤
　　④ バックエンドログで「PUTが一度も発行されていない」と確定
　　⑤ h2見出し確認のstrict modeエラーから
　　　「編集モードには正しく入れている」と判明
　　⑥ APIで直接確認し「サーバー側も未更新」と確定
　　⑦ セレクタの正規表現を見直し、「更新」ボタンの
　　　誤取得が根本原因と特定
　　→ Codex CLIの実装自体には最終的に欠陥がなかった
```

## 将来検討事項（実験終了時に再考）

### 6エージェント間の議論機能（保留）

【2026-06-21 検討・保留】実験Eの相互レビューやarticle-notesの
蓄積データを踏まえ、「6エージェントに直接議論させる」という
発案が出た。検討した内容を以下に記録し、実験終了時に再検討する。

```
検討した論点
└── 議論の形態
　　①エージェント同士が直接対話する（橋渡しが必要）
　　②Claudeが議論の構造を整理・創作する（既存データから）

└── 「同時実行」の可否
　　→ このClaudeチャット環境内では複数AIツールの並行呼び出しは
　　　不可能。各ツールは独立しており相互接続されていない。

└── 実現方法の候補
　　①複数ウィンドウを人間が手動で中継（橋渡し役）
　　②CLI系ツール（Claude Code/Codex CLI/Antigravity CLI）の
　　　非対話モード（ヘッドレス実行）をPythonスクリプトで連鎖
　　　させる方式 → 未検証（各ツールに-pオプション等の
　　　非対話モードがあるか確認が必要）
　　③共有Markdownファイル（discussion.md）を介した「擬似議論」
　　　→ IDE系（Antigravity IDE・Copilot Agent）も含めて
　　　　 全6エージェントが参加できる、最も実現性が高い方式
　　④MCP経由連携 → 各ツールがMCPサーバーとして機能する
　　　標準機能を持たないため、現実的に困難
　　⑤Zapier/n8n等のワークフロー自動化 → ローカル動作の
　　　エージェントのため組み込み不可

└── 「会話の相手をエージェントが決める」という動的指名方式も
　　検討した（各エージェントが回答の末尾で次の話者を指名し、
　　人間またはスクリプトがそれに従って次のツールを起動する）

└── リアルタイム性については、CLI系3エージェントのみ
　　スクリプトでの自動連鎖が技術的に最も近づける手段だが、
　　IDE系2エージェント（Antigravity IDE・Copilot Agent）は
　　人間の手動操作が必要なため、6エージェント全体としての
　　完全なリアルタイム議論は困難という結論に至った。

結論: 本実験では一旦保留とし、実験終了後（記事執筆フェーズの後）
に改めて実現方法を検討する。
```
