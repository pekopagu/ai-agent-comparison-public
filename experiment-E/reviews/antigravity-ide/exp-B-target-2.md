# コードレビュー結果: exp-B-target-2

## 1. バグ・潜在的な不具合
### テスト用DBファイル名の固定と並列実行時の競合リスク (重大度: medium)
`tests/test_api.py` の冒頭で `_TMP_DB = os.path.join(tempfile.gettempdir(), "tasks_test.db")` のように、一時ディレクトリ配下の固定のファイル名でテスト用データベースを定義しています。
もしテストを並列（例: `pytest-xdist` など）で実行した場合や、同一の開発・テストサーバー上で複数人が同時にテストを実行した場合、同一のDBファイルに対して書き込みや削除が競合し、`sqlite3.OperationalError` やデータの不整合によるテストのランダムな失敗が発生するリスクがあります。

---

## 2. セキュリティ上の問題
特になし。
`list_tasks` エンドポイントでの並び替えキー (`sort`) や順序 (`order`) について、ホワイトリストによる厳密な検証が行われているため、SQLインジェクションのリスクは適切に排除されています。また、検索クエリのバインドパラメータの受け渡しもプレースホルダーを利用して安全に行われています。

---

## 3. パフォーマンス上の問題
### データベース接続の都度開閉によるオーバーヘッド (重大度: medium)
`get_connection()` は呼び出されるたびに物理的な接続を開き、各APIの処理が終わると `finally` で閉じる実装となっています。同時アクセス数が少ないうちは大きな問題になりませんが、並行リクエストが増えた場合にはSQLiteへの接続開閉のボトルネックが発生し、パフォーマンスが著しく低下する可能性があります。

---

## 4. コードの可読性・保守性
### データベース接続管理が非標準的（Depends非使用） (重大度: medium)
FastAPIの標準的なベストプラクティスである `Depends`（依存性注入）を使用せず、APIエンドポイントの関数内で直接 `get_connection()` を呼び出して `try/finally` でクローズしています。これにより、テスト実行時にデータベースのパスや接続自体をモック化したり、一時DBに差し替えるための設定が難しくなっており、テストコード側で `import` 実行前に環境変数を無理やり書き換えるようなトリッキーな回避策が必要になっています。

---

## 5. ベストプラクティスへの準拠
### 静的ファイル用ディレクトリ指定のカレントディレクトリ依存 (重大度: low)
`main.py` の `_STATIC_DIR` は `__file__` を起点として絶対パスが計算されているため比較的安全ですが、`if _STATIC_DIR.exists():` で存在しない場合のチェックを入れている一方で、静的ディレクトリが存在しない場合でも警告が出ずにマウントをスキップする仕様になっています。デプロイ時の構成ミスなどに気づきにくくなる可能性があります。

---

## 6. テストの網羅性
### 正常系・異常系ともに網羅されているが、DBライフサイクルのテストが不足 (重大度: low)
テストコード自体はCRUD機能の正常系・異常系、フィルタ、ソート、検索、さらには静的ファイルの配信チェックまで非常に網羅的に書かれています。しかし、起動時の `lifespan`（DB初期化処理）が正しく機能するかどうかのテストは含まれていません。

---

## 改善提案

### 1. テスト用DBの一時ディレクトリとファイル名のユニーク化
`test_api.py` で固定ファイル名を使うのではなく、pytest の `tmp_path` フィクスチャ等を利用し、テストごとに隔離された一時ディレクトリ配下にDBファイルを自動生成するように変更します。

### 2. FastAPI の `Depends` を用いた接続管理へのリファクタリング
以下のように FastAPI の `Depends` を用いて接続を注入する設計に変更することで、テスト時に `dependency_overrides` を使って簡単に接続先を切り替えられるようになります。

```python
# app/database.py
def get_db():
    conn = get_connection()
    try:
        yield conn
    finally:
        conn.close()

# app/main.py
@app.get("/api/tasks")
def list_tasks(
    ...,
    conn: sqlite3.Connection = Depends(get_db)
):
    # エンドポイント内での try/finally による conn.close() は不要になる
```

## 総合評価
**7 / 10 点**

### 評価理由
Pydantic を用いた型安全なバリデーションや、SQLインジェクション対策がしっかりと施されており、全体として手堅く実装されています。テストの網羅性も高く、アプリケーション自体は正常に動作します。しかし、FastAPIの依存性注入の仕組みを活かしきれておらず、テストコードでのDBの差し替えに無理が生じている点、また並列テスト実行時の競合リスクが残っている点が評価を下げる要因となりました。
