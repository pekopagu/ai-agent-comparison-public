# コードレビュー結果: target-5

## 1. 問題点

### セキュリティ上の問題
*   **動的SQL構築によるSQLインジェクション脆弱性リスク (重大度: high)**
    *   `main.py` の `list_tasks` エンドポイントにおいて、`sql = f"SELECT * FROM tasks {where_sql} ORDER BY {sort_sql} {direction}, id {direction}"` のように、`sort_sql` や `direction` を直接文字列補完でSQLクエリに埋め込んでいます。Pydantic (Literal) によるバリデーションを通過する限りは実害はありませんが、型定義の変更やバリデーションの緩和が生じた場合に、容易にSQLインジェクション脆弱性を引き起こす極めて危険な設計です。

### パフォーマンス上の問題
*   **データベース接続の都度開閉による非効率性 (重大度: medium)**
    *   `sqlite3` を直接使用しており、クエリの実行ごとに `sqlite3.connect` を呼び出し、すぐにクローズしています。さらにタスクの作成・更新時には、書き込み処理と `fetch_task` による再取得処理で2回接続を開閉しており、オーバーヘッドが大きく非効率的です。コネクションプールを利用するか、リクエスト単位でセッションを共通化するべきです。
*   **サマリー表示のための過剰な API リクエスト (重大度: medium / low)**
    *   `index.html` ではサマリー表示を正確に行う（フィルタの影響を受けないようにする）ために、フィルタ付きのリクエストとは別に、毎回 `/tasks` （全件取得）を別途リクエストしています。これにより API リクエスト数が常に 2 倍になり、ネットワーク負荷が増加します。フロントエンド側のみでフィルタリングを行うか、バックエンド側で統計集計用のエンドポイントを提供するべきです。

### ベストプラクティスへの準拠
*   **冗長な DB 初期化の呼び出し (重大度: low)**
    *   `main.py` 内で、FastAPI の `lifespan` 起動時処理として `init_db()` を実行しているにもかかわらず、ファイル末尾のグローバルスコープ（142行目）でも `init_db()` を呼び出しています。二重に初期化が走るため、グローバルスコープでの呼び出しは不要です。

## 2. 改善提案

1.  **動的SQLの安全な構築 (`main.py`)**
    `ORDER BY` 句などのパラメータ化できない部分には、値がホワイトリスト（あらかじめ許可された識別子）に厳密に一致するかをバックエンドロジック側で明示的に検証した上で埋め込むようにします。
    ```python
    # 厳格なホワイトリスト検証の例
    if sort not in ("created_at", "due_date", "priority"):
        raise HTTPException(status_code=400, detail="Invalid sort key")
    if order not in ("asc", "desc"):
        raise HTTPException(status_code=400, detail="Invalid order")
    ```

2.  **接続プールの導入**
    リクエスト単位で同一の `Connection` オブジェクトを使い回すために、FastAPI の `Depends` などを利用して接続オブジェクトを管理するか、SQLAlchemy 等のプール機能を持つ ORM の使用を検討してください。

3.  **グローバルスコープでの DB 初期化の削除 (`main.py`)**
    ファイル末尾の `init_db()` （142行目）を削除し、`lifespan` 起動処理のみに統一します。

4.  **フロント側フィルタリングへの移行によるリクエスト削減 (`index.html`)**
    *   サマリー集計と一覧表示の双方を効率化するため、APIからは全件データを1回取得し、フロントエンド（Vue.jsの算出プロパティなど）でフィルタ処理を行う構成に切り替えることで、リクエスト数を削減できます。

---

## 3. 総合評価

### **6 / 10 点**
*   **評価理由**:
    SQLAlchemy などの ORM に依存せず、素の `sqlite3` と DDL で `CHECK` 制約を定義してデータ整合性を守っている点、またフロント側でサマリー集計がフィルタによって狂う問題を解決しようとする意識は評価できます。しかし、動的SQLの補完によるSQLインジェクションリスク、リクエスト毎の二重コネクション開閉、サマリー取得のための重複したAPIリクエストなど、アーキテクチャやパフォーマンスに重大な問題が見られます。
