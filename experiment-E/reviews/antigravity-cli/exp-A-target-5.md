# Target-5 コードレビュー結果

## 1. バグ・潜在的な不具合
* **重複したデータベース初期化（重大度: low）**
  * `main.py` において、Lifespan（スタートアップイベント）内の `init_db()` (line 17) と、モジュールロード時の `init_db()` (line 142) の2箇所で重複してテーブル初期化が実行されています。

## 2. セキュリティ上の問題
* **SQLクエリの直接文字列補間（重大度: medium）**
  * `list_tasks` エンドポイントにおいて、ソート条件（`sort_sql`）や順序（`direction`）をプレースホルダ（`?`）ではなく、`sql = f"... ORDER BY {sort_sql} {direction} ..."` のようにF文字列を用いてSQLクエリ内に直接埋め込んでいます。
  * 現在のコードではFastAPIの `Literal` 型アノテーションによる値の厳密なバリデーションが働いているため、外部からの不正文字列の注入（SQLインジェクション）は防がれていますが、プログラムの改修などでバリデーションが外れたり、あるいは型制約が緩んだりした場合に深刻なSQLインジェクション脆弱性へと発展する危険なコーディングパターンです。

## 3. パフォーマンス上の問題
* **プール化されない毎リクエストのSQLite接続・クローズ（重大度: high）**
  * `database.py` の `get_connection()` で `sqlite3.connect` を呼び出し、毎回クエリのたびに物理的なDBファイルのオープンとクローズを行っています。SQLAlchemyなどのORMやDB接続プール（Connection Pool）を利用していないため、APIの同時アクセスが発生した際に深刻なパフォーマンスボトルネックが発生します。
* **単一APIリクエスト内における多重のDB接続オープン（重大度: medium）**
  * 例えばタスクの更新 API `/tasks/{task_id}` (PUT) では、1つのリクエストを処理する間に `get_connection()` が最大 4回（存在チェック `fetch_task` ✕ 2回、更新実行 ✕ 1回、結果取得 `fetch_task` ✕ 1回）も個別に呼び出され、接続の開閉が繰り返されます。これによりSQLiteのファイルI/Oオーバーヘッドが極めて大きくなります。
* **フロントエンドでの非効率なAPI二重リクエスト（重大度: low）**
  * フロントエンドの `loadTasks` において、フィルタ結果のロード `this.tasks = await ...` と、統計計算用全件データのロード `this.allTasks = await ...` を順次同期的に2回リクエストしています。フロントエンドのアクションのたびにAPI接続が2往復するため、ネットワーク・サーバー負荷の両面で非効率です。

## 4. コードの可読性・保守性
* ORMを使用せず生SQLクエリをPythonコード内に文字列として埋め込んでいるため、SQLAlchemy等を使った実装と比較して、テーブル定義の変更（カラム追加等）時のメンテナンスコストが高くなります。

## 5. ベストプラクティスへの準拠
* 生の `sqlite3` ライブラリの直接操作、F文字列による動的クエリ組み立て、リクエスト毎の接続ブロッキングなど、現代の非同期FastAPI/Python Web開発におけるベストプラクティス（ORMやアシンクロナスドライバーの採用、プール管理）から大きく逸脱しています。

## 6. テストの網羅性
* `tests/test_api.py` にテストが記述されていますが、101行程度と少なく、詳細なソート挙動の分岐や空白値等の境界値検証が省略されています。
* テスト実行時に物理ファイル `test_tasks.db` をディスク上に書き出しており、テストの多重並行実行に対応できず、テスト完了時にファイルを消すクリーンアップ処理の管理が必要です。

---

## 改善提案
1. データベースアクセスライブラリとして SQLAlchemy または SQLModel 等のORMを導入し、コネクションプールによる再利用とスレッドセーフなセッション管理を行ってください。
2. 1つのリクエストを処理するスコープ（例: FastAPIのDepends）において、1つのDB接続（コネクション）およびトランザクションを共有・再利用するようにリファクタリングしてください。
3. `ORDER BY` のソート条件組み立てにおいて、事前に定義した安全なカラムマッピングオブジェクトから対応する値を取得する方式に変更し、SQLの動的文字列補間によるクエリ生成を排除してください。
4. テスト用DBをインメモリ SQLite (`:memory:`) に変更し、物理ファイルの生成を防止してください。

---

## 総合評価
**5.5 / 10 点**

FastAPIで開発する上でORMを使わず、かつ接続プールやリクエスト内での接続共有を行っていないため、著しいパフォーマンスの懸念があります。動的SQL生成によるセキュリティの懸念やフロントエンドの多重リクエストを含め、全体的に設計の洗練度が低いと言わざるを得ません。
