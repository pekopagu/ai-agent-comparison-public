# コードレビュー結果: target-3 (task-app)

## 1. 問題点

### Toggle処理における並行実行時のレースコンディション (重大度: medium)
- **詳細**: `repository.py` の `toggle_task` において、`get_task` で現在の `completed` 状態をメモリ上に読み出してから、その反転値を `UPDATE` 文で書き込んでいます。
- **影響**: 同時に同じタスクに対してトグルリクエストが発生した場合、状態の読み出しと書き込みの間に別のスレッドによる更新が割り込み、一方のトグル操作が無視される「ロストアップデート」が発生します。

### 接続ごとのディレクトリ作成によるオーバーヘッド (重大度: low)
- **詳細**: `database.py` の `connect` 内で、接続要求があるたびに `path.parent.mkdir(parents=True, exist_ok=True)` を実行しています。
- **影響**: APIリクエストの都度（`get_db` が呼ばれるたび）にOSのファイルシステムに対してディレクトリ作成チェックのシステムコールが発生し、不要なI/Oオーバーヘッドが生じます。

### データベース接続プーリングの欠如 (重大度: low)
- **詳細**: リクエストごとに `sqlite3.connect` を行い、コネクションプールを使用していません。
- **影響**: 高並行アクセス時に接続のオーバーヘッドが発生します。

### データベースインデックスの欠如 (重大度: low)
- **詳細**: テーブル設計において `completed`、`priority`、`due_date`、`created_at` にインデックスがありません。
- **影響**: タスク一覧の取得処理や統計情報の算出（`SUM(CASE...)` など）がフルスキャンになり、将来的にスロークエリ化する原因となります。

---

## 2. 改善提案

### トグル処理のアトミック化
- `completed` の反転更新はSQL単体で完結させ、アトミックに更新するよう修正してください：
  ```sql
  UPDATE tasks SET completed = 1 - completed, updated_at = ? WHERE id = ?
  ```

### ディレクトリ作成処理の実行タイミング修正
- `path.parent.mkdir` は、APIリクエストごとの接続処理（`connect`）から除外し、アプリケーション起動時に1度だけ呼び出される `init_db` 内に移動します。

### データベース接続の共有またはプーリング
- リクエストごとの接続コストを低減するため、スレッドセーフなコネクションプールを導入することを推奨します。

### インデックスの定義
- 検索やソート条件として多用される `completed`、`priority`、`due_date` などのカラムにインデックスを設定します。

---

## 3. 総合評価
**8.5 / 10**

FastAPIの `app.state` を利用してデータベースパスを安全に保持・切り替える仕組みや、テストでpytestの `tmp_path` を利用したDIによる上書きが実現できている点は非常に優秀です。リポジトリパターンを採用し、DB操作とAPIコントローラーが明確に分離されていて保守性が高いコードになっています。トグル操作のアトミック化や不要なI/O呼び出しの削減を行うことで、さらに洗練されたコードになります。
