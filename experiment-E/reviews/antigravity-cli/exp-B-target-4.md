# コードレビュー結果: target-4 (task-app)

## 1. 問題点

### 更新（PUT）時にDescriptionが初期化（nullクリア）できないバグ (重大度: high)
- **詳細**: `database.py` の `update_task` において、`description` フィールドの更新判定が `if description is not None:` となっています。
- **影響**: クライアントから「詳細説明（description）を削除して空にしたい」という意図で `{"description": null}` を送信した際、Pydanticスキーマは正常に処理されますが、`update_task` 内で `description is not None` が `False` になるため更新対象外となり、データベース上の既存の説明文が消去できません（一度設定した説明文を `null` にクリアすることが不可能になっています）。なお、`due_date` については `clear_due_date` フラグで対処されていますが、`description` にはそのようなケアが抜けています。

### Toggle処理における並行実行時のレースコンディション (重大度: medium)
- **詳細**: `database.py` の `toggle_task` は、`self.get_task(task_id)` を呼び出して現在の `completed` 状態をメモリに保持してから、`self.update_task` でその反対の値を書き込んでいます。
- **影響**: 同時に同一タスクへのトグル要求が発生した際に競合が発生し、一方の更新が失われる（ロストアップデート）原因になります。

### モジュールロード時のDB初期化副作用 (重大度: low)
- **詳細**: `main.py` のモジュールレベル（行29）で `_db = Database()` が実行され、そのコンストラクタ内で `_init_schema()` を通して本番DBファイル（`tasks.db`）が作成されています。
- **影響**: テストの実行時など、`main.py` をインポートしただけで自動的に本番用のデータベースファイルが生成されてしまう副作用があります。

### データベース接続プーリングの欠如 (重大度: low)
- **詳細**: 各メソッドで `with self._connect() as conn:` を使用し、毎回接続を開閉しています。
- **影響**: リクエストが多くなった場合に接続確立のオーバーヘッドが無視できなくなります。

### インデックスの欠如 (重大度: low)
- **詳細**: `tasks` テーブルに主キー以外のインデックスが存在しません。
- **影響**: タスク一覧の並び替え・絞り込み時にフルスキャンが発生し、パフォーマンスが劣化します。

---

## 2. 改善提案

### null更新（値の消去）に対応するよう更新ロジックを修正
- 更新用スキーマで値が意図的に `null` で送られてきた場合と、未指定（省略）の場合を区別し、データベースへ書き込む仕組みに修正してください。例えば、`update_task` に渡す引数をキーワード引数にするか、辞書（dict）として受け取り、キーが存在する場合にのみ更新処理を行う設計にします。

### トグル処理のアトミック化
- `toggle_task` をアトミックな1文の更新クエリに変更します：
  ```sql
  UPDATE tasks SET completed = 1 - completed, updated_at = ? WHERE id = ?
  ```

### 起動時のDB初期化の分離
- `Database` インスタンスの生成およびスキーマ構築（`_init_schema`）はモジュールレベルでの即時実行を避け、FastAPIの `lifespan` 起動イベント内などで明示的に実行する構成にします。

---

## 3. 総合評価
**7.5 / 10**

データベース接続と操作ロジックをオブジェクト指向設計（`Database` クラス）でカプセル化している点は、コードの可読性を高めており良好です。しかし、「descriptionをnullに更新できない」という致命的なAPIバグが残っていることや、トグル処理のレースコンディション、インポート時の副作用など、考慮漏れが見受けられます。
