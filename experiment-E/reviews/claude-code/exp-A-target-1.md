# コードレビュー: target-1 (Premium Task Manager)

対象: `targets-A/target-1/task-app`
構成: FastAPI + SQLAlchemy(ORM) + SQLite / Vue 3 (CDN, 開発版) 単一ファイル

---

## 概要
要件（CRUD・フィルタ・ソート・期限切れ表示・統計ダッシュボード）を一通り満たした動作する実装。フロントエンドのデザインが非常に作り込まれている。一方で、バックエンドのソート分岐がやや冗長で、フロントには大量のコメントの残骸があり保守性を下げている。

---

## 問題点

### High
特になし（致命的な不具合は見当たらない）。

### Medium
1. **`schemas.py` の入力値が strip されない（バリデーションの抜け）**
   `TaskBase.title` は `min_length=1` のみで、空白のみの文字列 `"   "` を許容してしまう。フロントの `validateForm()` は `trim()` で弾くが、API を直接叩くと空白タイトルが保存される。サーバ側で `field_validator` による strip/空判定を入れるべき（target-3/5 は対応済み）。

2. **統計取得のための二重リクエスト**
   `updateAll()` が `fetchTasks()`（フィルタ付き）と `calculateStats()`（フィルタ無し全件）の 2 リクエストを毎回発行する。`calculateStats` 内で `isOverdue` 集計のため全件を取得しているが、フィルタ操作のたびに全件取得が走る。件数が増えると無駄が大きい。統計用エンドポイント（`GET /tasks/stats` で集計）を用意するか、全件を一度だけ取得してクライアント側でフィルタ/ソートする設計が望ましい。

### Low
3. **`main.py` のソート分岐が冗長**
   `order==desc/asc` × `sort==due_date/その他` の入れ子分岐になっている。`asc()/desc()` と NULLS LAST 用の式を共通化すると見通しが良くなる。

4. **`case({...}, value=...)` の辞書呼び出し形式**
   SQLAlchemy 2.0 では `case()` の whens は位置引数（タプル列挙）が推奨。辞書 + `value=` 形式は当面動作するが、将来的な非推奨化リスクがある。

5. **フロントに大量のコメント残骸（`index.html` 1122–1140 行付近）**
   設計検討の思考過程がそのまま長文コメントとして残っており、可読性を著しく損ねている。削除すべき。

6. **`datetime.now()` がナイーブなローカル時刻**
   タイムゾーン非対応。`created_at` を UTC で保持する方が堅牢（target-3 は UTC 保持）。

7. **日付パースのタイムゾーンずれ**
   フロントの `formatDate(new Date("2026-06-30"))` は UTC 0時として解釈されるため、負のオフセット環境では 1 日前に表示されうる。`isOverdue` は文字列比較で回避しているが `formatDate` は未対応。

---

## 改善提案
- `title`/`description` に `field_validator` を追加し、サーバ側で strip と空チェックを行う。
- 統計は専用集計エンドポイント化、または全件取得＋クライアント集計に統一し、リクエスト数を削減。
- `index.html` の検討コメントを削除し、JS をコンポーネント/メソッド単位に整理。
- ソート処理を `order_by` ビルダー関数に抽出して重複を解消。
- `created_at` を `datetime.now(timezone.utc)` 基準に。

---

## テストの網羅性
`tests/test_api.py` は作成・バリデーション(422)・フィルタ・ソート(priority/due_date)・取得・更新・削除・404 をカバーし良好。ただし以下が未テスト:
- 空白のみタイトル（上記 Medium 1 のため現状は通ってしまう）
- 部分更新（PUT で一部フィールドのみ）の挙動
- 一覧の空配列ケース
- `due_date` が NULL の混在ソート

---

## 総合評価: **7 / 10**
動作・要件充足・UI 完成度は高い。減点要因はサーバ側の空白バリデーション欠落、統計の二重リクエスト、そして本番コードに残った大量の検討コメント。これらを整理すれば 8〜9 相当。
