# コードレビュー: target-3

対象: `targets-A/target-3/task-app`
構成: FastAPI + SQLAlchemy 2.0(ORM, Mapped) + SQLite / Vue 3 (CDN)

---

## 概要
5 つの中で最も洗練された実装。SQLAlchemy 2.0 のモダン記法（`Mapped` / `select` / `db.get` / `db.scalars`）を一貫して使用し、Enum による型安全なクエリパラメータ、`get_task_or_404` ヘルパによる DRY、インデックス付与、UTC 保持、`field_validator` による strip と、バックエンドの品質が高い。フロントもシンプルで堅実。

---

## 問題点

### High
特になし。

### Medium
1. **PUT が全項目置換（部分更新不可）**
   `TaskUpdate(TaskBase)` を継承しているため、PUT は全フィールド必須の完全置換セマンティクス。`title` 未送信時は 422 になる。フロントは常に全項目を送るため整合は取れているが、API 単体では「ステータスだけ更新」ができない。仕様上 PUT=全置換と割り切るなら問題ないが、部分更新が必要なら `PATCH` か `exclude_unset` 版の用意を検討。明示的な設計判断としてドキュメント化すべき。

### Low
2. **`priority_rank` を毎クエリ構築**
   ソートが priority 以外でも `case` 式を生成している。実害はほぼ無いが、`sort == priority` の場合のみ構築すれば無駄がない。

3. **サマリーがフィルタ後リストから算出**
   `summary` を `tasks.value`（フィルタ適用済み）から計算しているため、ステータスでフィルタすると集計値もそのフィルタ結果に連動する。ダッシュボードとして「全体の件数」を出したい場合は意図とズレる。期限切れ件数の集計も summary に無い点が target-1 と比べ簡素。

4. **`created_at` のローカル日付ずれ（フロント表示）**
   target 共通の軽微な懸念。`isOverdue` は `T00:00:00` を付けてローカル解釈しており配慮あり（良い）。

---

## セキュリティ / パフォーマンス
- ORM パラメータバインドにより SQL インジェクションなし。
- `status/priority/sort/order` がすべて Enum 検証され、不正値は自動的に 422。OpenAPI ドキュメントにも反映される（best practice）。
- `status` / `priority` / `due_date` / `created_at` にインデックスが付与されており、フィルタ・ソートのスケール性が良い。
- CORS はオリジン限定済みで妥当。

---

## テストの網羅性
作成・取得・バリデーション（空白タイトル/不正ステータス）・更新（全置換と None 化）・削除(204)・フィルタ・ソート(due_date/priority)・404 をカバー。フィクスチャは `StaticPool` のインメモリ DB をテストごとに再生成しており分離性も良好。
未カバー: 部分更新の不可挙動（上記 Medium 1）、優先度フィルタ単独、空一覧。総合的には十分。

---

## 改善提案
- PUT のセマンティクス（全置換）を README / docstring に明記、もしくは PATCH を追加。
- ダッシュボード集計を要件に合わせ「全件ベース」にするか、現状の「フィルタ連動」を意図として明記。
- `case` 構築を priority ソート時のみに限定。

---

## 総合評価: **9 / 10**
モダンで型安全、DRY、インデックス・UTC・strip まで配慮が行き届いた模範的実装。減点は PUT の部分更新非対応という設計上の割り切りのみ。5 ターゲット中の最高品質。
