# target-2 コードレビュー

## 問題点

- 重大度: medium - `app/main.py:169-189` の PUT は未指定フィールドを保持しますが、更新フィールドが空の場合は `updated_at` も更新されず、そのまま既存行を返します。空 PUT を許容する仕様ならよいものの、クライアントからは成功に見えるため監査や同期の意図が曖昧です。
- 重大度: medium - `app/main.py:102-108` の検索は `LIKE` をそのまま使うため、ユーザー入力中の `%` と `_` がワイルドカードとして解釈されます。通常の全文検索としては意図しない大量一致になり得ます。
- 重大度: low - `app/main.py:119-129` の priority 並び替えは `desc` 指定時に high が先、`asc` 指定時に low が先になりますが、API 利用者から見た優先度順の意味が明記されていません。UI と API の期待がずれる可能性があります。
- 重大度: low - `app/database.py:20` の DB パスは環境変数で上書き可能ですが、SQLite 接続に `timeout` や WAL 設定がなく、同時書き込みが増えると `database is locked` が起きやすい構成です。
- 重大度: low - コメント・docstring が広範囲に文字化けしており、`app/main.py`、`app/database.py`、`app/schemas.py` の保守性を大きく下げています。コード自体はコンパイル可能でしたが、レビューや障害対応時の理解コストが高いです。
- 重大度: low - テストは CRUD、異常系、フィルタ、ソート、静的配信まで広くありますが、ワイルドカード検索、同時更新、空 PUT、DB ロック、フロント操作の回帰は未検証です。

## 改善提案

- 空 PUT は 400/422 にするか、`updated_at` を更新するか、仕様として README/API ドキュメントに明記する。
- 検索文字列中の `%` と `_` をエスケープする。必要なら別途「ワイルドカード検索」機能として明示的に提供する。
- SQLite は `timeout`、WAL、短いトランザクション設計を導入し、将来の並行アクセスに備える。
- 文字化けしたコメント・docstring・テスト名周辺の文字列を UTF-8 で修復し、エディタ設定も統一する。
- テストに検索特殊文字、空更新、同時書き込みに近いケース、フロントの主要操作を追加する。

8/10
