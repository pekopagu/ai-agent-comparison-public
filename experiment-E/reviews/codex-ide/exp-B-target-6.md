# target-6 コードレビュー

## 問題点

- 重大度: high - `database.py:4` の DB URL が `sqlite:///./tasks.db` 固定で、環境変数による切り替えがありません。テストは `test_main.py:11-32` で依存関係を上書きしていますが、本番・開発・CI で DB を分離しにくく、起動ディレクトリ次第で別 DB が作られます。
- 重大度: medium - `main.py:78-85` の DELETE は 200 OK と JSON ボディを返します。他 target や一般的な REST 慣習では削除成功は 204 が自然で、フロントやテスト以外のクライアントと契約がずれやすいです。
- 重大度: medium - `main.py:32` の検索は SQLAlchemy の `contains(q)` を使っており、LIKE の `%` と `_` がワイルドカードとして効きます。ユーザーが通常文字として検索したい場合に意図しない一致になります。
- 重大度: medium - `schemas.py:7-17` の `due_date` は正規表現で形だけを確認しており、`2026-99-99` のような存在しない日付を受け付けます。後段の期限判定や UI 表示で破綻します。
- 重大度: medium - `main.py:58-75` は `exclude_unset=True` で部分更新できますが、`due_date=None` を送ると期限をクリアできます。一方で `description=None` はスキーマ上許可されず、空文字にする必要があります。仕様としては成立しますが、フロント/API 利用者にとって一貫性が弱いです。
- 重大度: low - `main.py:91` の静的配信パスが `"static"` の相対パスです。起動ディレクトリが違うとフロントが配信されません。
- 重大度: low - テストは CRUD、バリデーション、フィルタを押さえていますが、DB パス切り替え、無効な実在しない日付、検索特殊文字、204 契約、静的配信、フロントのドラッグ更新は未検証です。

## 改善提案

- DB URL は `DATABASE_URL` などの環境変数で切り替え可能にし、既定値はアプリルート基準の絶対パスにする。
- DELETE は 204 No Content に揃えるか、200 JSON を正式仕様として README とクライアント契約に明記する。
- `due_date` は `datetime.date` 型または Pydantic validator で実在日付として検証する。
- LIKE 検索の特殊文字をエスケープし、必要なら通常検索とワイルドカード検索を分ける。
- 静的配信は `Path(__file__).resolve().parent / "static"` など絶対パスに変更する。
- テストに DB 設定、日付妥当性、検索特殊文字、静的ファイル配信、ドラッグによる status 更新相当のケースを追加する。
